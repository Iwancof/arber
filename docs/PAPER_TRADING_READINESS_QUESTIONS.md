# Paper Trading 移行に向けた設計確認

**宛先**: 設計担当者
**発信**: 実装チーム
**日付**: 2026-04-07
**件名**: Paper Trading 開始前に確認すべき3つの設計判断

---

## 背景

パイプラインが v2 プロンプトで安定稼働しています（Noise classifier 含む、エラー 0）。
次のステップとして Paper Trading（Alpaca Paper API）に進みたいのですが、実データで検証したところ **3つの設計上の不備** が見つかりました。

---

## 問題1: スコア計算式が保守的すぎて取引シグナルが出ない

### 現象

v2 プロンプトの出力は高品質ですが、Policy Engine のスコア計算を通すと **全て no_trade** になります。10件の判断で **long_candidate が一度も出ていません**。

```
Case             conf   1d_p   5d_p    score   action
─────────────────────────────────────────────────────
Strong signal    0.67   0.64   0.61    0.273   no_trade
Clear signal     0.72   0.68   0.63    0.349   no_trade
Very strong      0.78   0.72   0.68    0.444   wait_manual
Moderate         0.59   0.58   0.55    0.145   no_trade
Noise            0.51   0.51   0.50    0.015   no_trade
```

**Thresholds**: LONG = 0.60, MANUAL = 0.40

confidence 0.78 + p_outperform 0.72 という**非常に強いシグナル**でも score = 0.44 にしかならず、`long_candidate` (0.60) に到達しません。

### 原因

現在のスコア計算式:
```python
base_score = (confidence - 0.5) * 2        # [0,1] → [-1,1]
horizon_score = Σ (p_outperform - 0.5) * 2 * weight
combined = (base_score + horizon_score) / 2  # ← ここで半減
```

問題は **2回の半減** です：
1. `(confidence - 0.5) * 2` で confidence 0.78 → base 0.56
2. `(base + horizon) / 2` でさらに半減 → 最終 0.44

### 確認事項

**Q1-1**: スコア計算式をどう修正すべきですか？以下の選択肢があります：

**案A**: 閾値を下げる（LONG = 0.35, MANUAL = 0.25）
- メリット: 既存ロジック変更最小
- デメリット: 閾値の意味が直感的でなくなる

**案B**: スコア式を修正して [-1, 1] の全レンジを使う
- 例: `combined = base_score * 0.4 + horizon_score * 0.6`（半減しない）
- メリット: confidence 0.78 + p_out 0.72 → score ~0.60 で long_candidate になる

**案C**: v2 プロンプトの confidence バンドに合わせたスコアマッピング
- 例: confidence_after > 0.65 かつ p_outperform_1d > 0.60 → 自動的に long_candidate
- メリット: プロンプト設計のキャリブレーションと直結

**案D**: 他の設計

**Q1-2**: v2 プロンプトの confidence bands（0.50-0.52 = no edge, 0.65-0.72 = clear edge）と Policy Engine の閾値の対応関係を定義してください。

## 問題2: エグジット（手仕舞い）戦略がない

### 現象

現在の設計は **エントリーシグナル（買い/売り判断）のみ** です。ポジションを持った後、いつ手仕舞うかの仕組みがありません。

### 確認事項

**Q2-1**: v1 Paper のエグジット方式はどれですか？

**案A**: ホライズンベース自動クローズ
- 1d シグナルで入ったら翌営業日クローズ
- 5d シグナルで入ったら5営業日後クローズ
- メリット: シンプル、予測と検証が一致

**案B**: 反転シグナルでクローズ
- 同一銘柄で逆方向のシグナルが出たらクローズ
- メリット: 市場変化に適応

**案C**: 固定ストップ/テイクプロフィット
- 例: -2% ストップ、+5% テイクプロフィット
- メリット: リスク管理が明確

**案D**: 手動のみ
- Paper Phase では人間が `/ops-apply` でクローズ指示
- メリット: 最もシンプル、学習に適する

**案E**: 他の設計

**Q2-2**: 同一銘柄に複数シグナルが来た場合のルールは？
- 既存ポジションがあるときに同方向シグナル → 増し玉？無視？
- 既存ポジションがあるときに逆方向シグナル → クローズ？両建て禁止？

## 問題3: ポジションサイジングが未定義

### 現象

`decision.size_cap` が全件 `None` です。long_candidate が出ても発注数量が決まりません。

### 確認事項

**Q3-1**: Paper Phase のポジションサイジングはどうしますか？

**案A**: 固定金額
- 例: 1シグナルあたり $1,000
- Paper account は通常 $100,000 でスタート

**案B**: ポートフォリオ比率
- 例: 1シグナルあたりポートフォリオの 2%
- confidence が高いほど大きく（1%〜3%）

**案C**: 固定株数
- 例: 1シグナルあたり 10 株
- 最もシンプル

**案D**: score ベース
- score 0.60 → 1%, score 0.80 → 2% のように比例

**Q3-2**: 最大同時ポジション数に制限はありますか？
- 例: 最大5銘柄まで、1銘柄あたりポートフォリオの5%まで

## 参考: Paper Trading に必要な実装（設計確認後）

上記3点が決まれば、以下を実装してPaper稼働できます：

1. **AlpacaPaperBrokerAdapter** — Alpaca Paper API で注文送信
2. **スコア計算修正** — Q1 の回答に従って Policy Engine を調整
3. **エグジットロジック** — Q2 の回答に従って実装
4. **サイジングロジック** — Q3 の回答に従って `size_cap` を計算
5. **パイプライン接続** — decision → submit_order の配線
6. **execution_mode = paper** への切り替え

技術的にはすぐ実装可能です。設計判断のみお待ちしています。
